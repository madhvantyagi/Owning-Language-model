Part 1 — What the Hell Is a Logit?

This is the single most important concept to understand, because everything — evaluation, loss, generation, perplexity — is built on top of logits. If you don't understand logits, sections 8 and 9 will always feel like vague hand-waving.

The Fundamental Problem a Language Model Solves

A language model has one job: given some tokens it has seen so far, predict which token comes next.

Our model has a vocabulary of 32,768 tokens. So at each position in a sequence, the model needs to answer:

"Out of these 32,768 possible tokens, how strongly do I believe each one should be the next token?"

That's it. That's the whole thing. Every other detail — attention, MLP, layers, embeddings — is just machinery to make this single answer better.

The Raw Answer: A Vector of 32,768 Numbers

After all 30 layers of computation, the model outputs a single vector for each position in the sequence. This vector has exactly one number per vocabulary item:

At position 2 (predicting after "sat"): logits[0, 2] = [ -1.24, 0.87, 5.31, -3.11, ..., 2.04 ] ↑ ↑ ↑ ↑ ↑ token token token token token 0 1 2 3 32767 Shape: (32768,) — one raw score per vocab item

These raw numbers are called logits. They are unbounded real numbers — they can be positive, negative, large, or small. They are not yet probabilities. They are raw scores.

Key fact — Logits are unnormalized

Logits can be anything from -∞ to +∞. A logit of 5.3 for token 2 means "the model has some evidence for token 2." A logit of -3.1 means "the model has evidence against token 3." But you cannot compare logits across different contexts — a 5.3 in one position means something different from a 5.3 in another.

Where Do Logits Come From? (The LM Head)

The very last operation in the model is the LM Head. It's a single matrix multiplication: 

# x is the final hidden state, shape (B, T, 960)
# embed.weight is the embedding matrix, shape (32768, 960)
logits = x @ embed.weight.T    # shape (B, T, 32768)

Each of the 960 hidden dimensions votes, with some weight, for or against each vocabulary token. The LM head aggregates these 960 votes into one score per token.

Perspective 1 — Linear Algebra

Think of the 960-dim hidden vector as a point in a 960-dimensional space. The embedding matrix stores 32,768 "reference points" — one for each token. The LM head computes the dot product (similarity) between the hidden vector and each reference point. A high dot product means "this hidden state is similar to that token's embedding, so that token is a likely continuation."

Perspective 2 — Geometry

Imagine a 2D space (instead of 960D). Each vocabulary item is a point in this space. The hidden state is another point. The logit for each vocabulary item is proportional to the cosine of the angle between the hidden state and that item's point, scaled by their lengths. Items pointing in the same direction as the hidden state get high logits; items pointing opposite get low (negative) logits.

Perspective 3 — Voting / Analogy

Imagine 960 experts, each with an opinion on which token should come next. Each expert has a "vote direction" — they support some tokens and oppose others. The hidden state is the consensus of all 960 experts. The LM head counts the votes: "Expert 1 mildly supports token 464, expert 2 strongly opposes token 17, ..." The logit is the net vote tally for each token.

Memory aid — Logits are "logarithmic odds units"

The name "logit" comes from "logistic unit." In statistics, the logit function is the inverse of the logistic (sigmoid) function: \(\text{logit}(p) = \log(p/(1-p))\). A logit of +∞ means "I'm certain this token is next." A logit of -∞ means "I'm certain this token is NOT next." A logit of 0 means "I'm equally sure for and against."

Why Not Just Output Probabilities Directly?

This is a natural question. Why go through logits instead of just outputting probabilities (numbers between 0 and 1 that sum to 1)?

Because it's much easier for the model to learn.

Probabilities are constrained: they must be non-negative and sum to 1. If the model output probabilities directly, the last layer would need to enforce these constraints — incredibly hard for a neural network. Logits have no constraints. The model can output any real number for any token. We convert logits to probabilities later using softmax.

This is the most liberating design decision in ML

The model never has to worry about creating valid probabilities. It just outputs raw scores (logits). The softmax function, applied after the model, handles the conversion. This separation lets the model focus on representation learning — figuring out which tokens are good continuations — without worrying about normalization.

◈ ◈ ◈

From Logits to Probabilities: Softmax

Softmax takes the raw logits and converts them to a valid probability distribution. It does two things:

  1. Exponentiate — turn every logit into a positive number
  2. Normalize — divide by the sum so they add up to 1

For a single position with logits \(z_1, z_2, \dots, z_{32768}\):

\[ p_i = \frac{e^{z_i}}{\sum_{j=1}^{32768} e^{z_j}} \]

After softmax, each \(p_i\) is between 0 and 1, and \(\sum p_i = 1\). This is a valid probability distribution over the vocabulary.

Perspective 1 — Why Exponentiate?

The exponential function \(e^z\) turns any real number into a positive number. If logit = 5.3 → \(e^{5.3} \approx 200\). If logit = -3.1 → \(e^{-3.1} \approx 0.045\). Large positive logits become large weights; negative logits become tiny weights. The exponential preserves order — if logit A > logit B, then \(e^A > e^B\).

Perspective 2 — Why Not Just Take Absolute Value or Square?

Good question! Both would make all logits positive. But:

  • Absolute value destroys information — -5 and +5 become the same value, but they mean opposite things (strong evidence against vs. strong evidence for).
  • Square also destroys sign information and exaggerates large values disproportionately.
  • Exponential preserves the order, keeps the ratio structure (difference in log space = ratio in probability space), and has nice gradients for optimization.
Perspective 3 — Information Theory

The logit \(z_i\) can be interpreted as the log-odds (logarithm of odds) for token \(i\) relative to some baseline. When we exponentiate and normalize, we're essentially computing:

\[ p_i = \frac{\text{odds}_i}{\sum \text{odds}_j} \]

where \(\text{odds}_i = e^{z_i}\). This is the natural way to convert "evidence scores" (logits) into "belief probabilities" (softmax).

Concrete Example with the Running Example

Our model sees "The cat sat" (3 tokens). At position 2 (predicting what comes after "sat"), the model outputs logits:

Check your understanding

At random initialization, all logits are roughly the same magnitude (drawn from a normal distribution with mean 0, std 0.02). They're all close to 0. After softmax, what probability does each token get? Approximately \(1/32768 \approx 0.00003\). The model is guessing uniformly. This is why initial loss ≈ \(\log(32768) = 10.40\) — we'll connect this in Part 2.

Part 2 — From Logits to Loss:
How "Wrong" Is the Model?

Now we understand: the model outputs logits, we softmax them into probabilities. But how do we measure whether the model is doing well? This is where cross-entropy loss comes in.

The Core Question

We have the true next token (from the training data). The model assigned some probability to it (via softmax of logits). How wrong was the model?

Intuition: If the model assigned 99% probability to the correct token, it was barely wrong — loss is tiny. If it assigned 0.001% probability, it was very wrong — loss is huge.

The Answer: Negative Log Probability

For a single prediction, the loss is:

\[ \text{loss} = -\log(p_{\text{true}}) \]

where \(p_{\text{true}}\) is the softmax probability of the correct next token.

Perspective 1 — Why Negative Log?

Let's reason from first principles. We want a function that measures "how wrong" the model is. What properties should it have?

  • If the model is perfectly confident and correct (\(p = 1.0\)), loss should be 0 (no penalty).
  • If the model is less confident (\(p = 0.5\)), loss should be positive and larger.
  • If the model is very wrong (\(p = 0.001\)), loss should be very large.
  • Loss should always be non-negative.

The function \(-\log(p)\) satisfies all of these:

  • \(-\log(1.0) = 0\) — perfect prediction, zero loss
  • \(-\log(0.5) = 0.69\) — moderate loss
  • \(-\log(0.001) = 6.91\) — large loss
  • As \(p \to 0\), \(-\log(p) \to \infty\) — infinite penalty for being infinitely wrong
Perspective 2 — Information Theory

In information theory, \(-\log_2(p)\) is the number of bits of surprise when an event with probability \(p\) occurs. If the model thought " on" had 99% probability, it takes very few bits to encode the surprise that " on" actually occurred. If it thought "elephant" had 0.001% probability, it takes many bits.

Cross-entropy loss = expected number of bits needed to encode the true tokens using the model's predicted distribution. Lower = better compression = better model.

At initialization (uniform probability \(1/32768\)), each token needs \(\log_2(32768) = 15\) bits to encode. That's why initial loss = 10.40 (in nats; in bits it would be 15).

Perspective 3 — Optimization (Why This Loss?)

This loss is equivalent to maximum likelihood estimation. We're finding the model parameters that maximize the probability of the actual training data. Minimizing \(-\log(p_{\text{true}})\) is the same as maximizing \(p_{\text{true}}\), which is the same as maximizing the likelihood of the data.

Why not use \(1 - p_{\text{true}}\) as the loss? Because it doesn't penalize small probabilities enough — the gradient would be tiny for very wrong predictions, and the model would never learn to avoid catastrophic mistakes.

The Full Batch Loss

In practice, we don't compute loss for just one position. We compute it for every position in every sequence in the batch.

For a batch of \(B\) sequences, each of length \(T\):

\[ \text{total\_loss} = \frac{1}{B \cdot T} \sum_{b=1}^{B} \sum_{t=1}^{T} -\log\big(p_{\text{true}_{b,t}}\big) \]

This is the average negative log probability across all \(B \times T\) predictions. We average (rather than sum) so the loss doesn't depend on the batch size.

Wait — what about positions where the model shouldn't predict?

In practice, some positions are padded or should be ignored. PyTorch's cross_entropy has an ignore_index parameter. We set ignore_index=-100. Positions with label -100 are excluded from the loss computation. This is standard.

Concrete Loss Calculation with Our Example

Let's trace through the actual loss computation:

# Input: "The cat sat" = [464, 3797, 6471]
# Labels:               [3797, 6471,  ???]  (shifted by 1 position)
#   The model sees "The" → should predict "cat"
#   The model sees "The cat" → should predict "sat"
#   The model sees "The cat sat" → should predict the next token

# Suppose the actual next token after "sat" is "on" (token 262)
# Labels = [3797, 6471, 262]

# For position 0 (predicting "cat"):
#   Model outputs logits[0]
#   softmax → probability of token 3797 ("cat") = 0.52
#   loss[0] = -log(0.52) = 0.65

# For position 1 (predicting "sat"):
#   softmax probability of token 6471 ("sat") = 0.31
#   loss[1] = -log(0.31) = 1.17

# For position 2 (predicting "on"):
#   softmax probability of token 262 ("on") = 0.28
#   loss[2] = -log(0.28) = 1.27

# Total loss = (0.65 + 1.17 + 1.27) / 3 = 1.03

# At initialization: every token gets probability ~1/32768
#   loss[each] = -log(1/32768) = log(32768) = 10.40
#   Total loss = 10.40 ← this is the starting point

As training progresses, the model gets better at assigning high probability to the correct next token → loss drops from 10.40 toward ~3.5 (for a good small model).

The mathematical chain in one line
\[ \text{input\_ids} \xrightarrow{\text{model}} \text{logits} \xrightarrow{\text{softmax}} \text{probabilities} \xrightarrow{-\log(p_{\text{true}})} \text{loss} \xrightarrow{\text{backward}} \text{gradients} \xrightarrow{\text{optimizer step}} \text{better model} \]

Every component in this chain must be differentiable so gradients can flow backward. Logits, softmax, and negative log probability are all designed to play nicely with calculus.

Where the Actual Cross-Entropy Formula Comes From

In PyTorch, F.cross_entropy(logits, labels) does both softmax and negative log loss in one fused operation. It's equivalent to: 

import torch.nn.functional as F

# These two are equivalent:
loss1 = F.cross_entropy(logits, labels)       # fused, numerically stable

probs  = F.softmax(logits, dim=-1)             # separate steps
loss2  = F.nll_loss(torch.log(probs), labels)  # nll = negative log likelihood

# loss1 == loss2 (minus tiny floating-point differences)

Why the fused version? If we do softmax separately and then take log, we can get \(\log(0) = -\infty\) for very small probabilities. The fused version computes \(\log(\text{softmax})\) directly using the identity:

\[ \log(\text{softmax}(z_i)) = z_i - \log\left(\sum_j e^{z_j}\right) \]

This avoids ever computing the probabilities explicitly, which is numerically stable even when some probabilities would underflow to 0.

Numerical stability is not a detail — it's the difference between training and NaN

If you implement softmax naïvely as exp(z) / sum(exp(z)), then for logit = 1000, \(e^{1000}\) overflows to infinity. The numerically stable version subtracts the max logit first: \(\text{softmax}(z_i) = e^{z_i - \max(z)} / \sum e^{z_j - \max(z)}\). PyTorch's fused cross_entropy handles all of this.

Part 3 — Section 8 Deep Dive:
Evaluation & Perplexity

Now we have all the pieces. Let's look at Section 8 of the architecture walkthrough and unpack every line.

What Is Evaluation?

During training, we track loss on the training data. But the model could memorize the training data without learning general patterns. So we periodically compute loss on a held-out validation set — data the model has never seen during training.

If training loss goes down but validation loss goes up → overfitting (memorizing). If both go down → genuine learning.

Validation Loss — What the Code Actually Does

From the architecture doc:

for _ in range(n_sequences):
    seq = next(stream)                                    # get a sequence
    input_ids = torch.tensor([seq[:seq_len]])             # (1, 2048)
    labels = torch.tensor([seq[1:seq_len+1]])             # (1, 2048) shifted
    outputs = model(input_ids=input_ids, labels=labels)   # forward + loss
    total_loss_sum += outputs.loss.item() * seq_len       # accumulate

Line by line:

LineWhat's happening
seq = next(stream) Get a sequence of tokens from the validation data stream. Shape: (2049,) — one extra because we need input + label.
input_ids = [seq[:2048]] First 2048 tokens are the input to the model.
labels = [seq[1:2049]] Tokens 1 through 2048 are the labels — what the model should predict at each position. This is the "shifted" version.
outputs = model(...) Run the forward pass. The model computes logits → softmax → cross-entropy with labels → returns a loss value (scalar).
total_loss_sum += loss * seq_len Accumulate the un-averaged loss (multiply by sequence length to undo the averaging). We'll divide at the end.

Why multiply by seq_len? The loss from the model is already averaged over all T positions. To sum across multiple sequences, we need to "un-average" each one: \(\text{sum} = \text{avg_loss} \times T\).

Why not just average the losses directly? Sequences might have different lengths (they don't in this case, but as a general pattern, weighting by length gives each token equal importance rather than each sequence equal importance).

Perplexity — The Most Intuitive Metric

Perplexity is just the exponential of the loss:

\[ \text{Perplexity} = e^{\text{loss}} \]

That's it. That's the whole formula. But what does it mean?

Perspective 1 — "Effective Vocabulary Size"

Perplexity answers: "How many tokens is the model effectively choosing among at each position?"

If perplexity = 1: the model is perfectly certain — it effectively chooses among 1 token (the correct one).

If perplexity = 32768: the model is guessing uniformly at random from the entire vocabulary.

If perplexity = 30: the model is "as uncertain as if it were choosing uniformly among 30 equally-likely tokens." Much better than 32,768, but still uncertain.

Perspective 2 — Information Theory

Perplexity is \(2^{H}\) where \(H\) is the entropy in bits. It's the branching factor of the language. If English has a perplexity of 30, that means "on average, the next word is one of 30 possible choices, as far as the model is concerned."

A human has perplexity around 10-20 on typical text. GPT-4 achieves around 10-15. Our small model targets 20-35.

Perspective 3 — Gambling / Prediction

Imagine you're betting on the next token. Your wealth grows by a factor of \(1/p_{\text{true}}\) each round (you bet everything on the token you think is most likely). After many rounds, your wealth grows roughly as \((1/\text{perplexity})\) per token. Lower perplexity → faster wealth growth → better predictor.

The Perplexity Table, Explained

PerplexityWhat it means concretely
1.0 Perfect prediction. The model is always 100% certain and correct. Impossible in practice — language is inherently unpredictable.
32,768 Uniform random guessing. Loss = log(32768) = 10.40. This is where the model starts at initialization.
~500 A model that's learned some basic patterns (common words, sentence structure) but is still very uncertain. Loss ≈ 6.2.
~50 A decent model. It understands grammar and common word patterns. Loss ≈ 3.9.

Why perplexity is more intuitive than loss

Loss = 3.42 is hard to interpret. But perplexity = \(e^{3.42} = 30.6\) means "the model is as uncertain as if it were choosing among 30 equally-likely tokens." That's something you can wrap your head around.

Generation Test — Using the Model After Training

Evaluation isn't just about numbers — we also want to see what the model actually generates. This is the generation test. 

def generate(model, input_ids, max_new_tokens=50, temperature=0.8, top_k=50):
    for _ in range(max_new_tokens):
        # 1. Get logits for the last position only
        logits = model(input_ids).logits[:, -1, :]     # (1, 32768)

        # 2. Apply temperature scaling
        logits = logits / temperature

        # 3. Apply top-k filtering
        top_k_values, _ = torch.topk(logits, top_k)
        logits[logits < top_k_values[:, -1]] = -inf

        # 4. Convert to probabilities and sample
        probs = F.softmax(logits, dim=-1)
        next_id = torch.multinomial(probs, num_samples=1)

        # 5. Append to sequence
        input_ids = torch.cat([input_ids, next_id], dim=1)

    return input_ids

Why only the last position? During generation, we already know the entire prompt. We only care about what comes next. The model computes logits for all positions (it's more efficient), but we only look at logits[:, -1, :] — the last position's prediction.

Temperature — Controlling Randomness

Before softmax, we divide logits by a temperature \(T\):

\[ p_i = \frac{e^{z_i / T}}{\sum_j e^{z_j / T}} \]
TemperatureEffect on softmaxGeneration style
\(T \to 0\) Probability mass concentrates on the token with highest logit. Softmax becomes argmax. Deterministic, repetitive, safe
\(T = 1.0\) No change — standard softmax. Balanced — what the model naturally learned
\(T > 1.0\) Probabilities become more uniform. Low-probability tokens get a bigger chance. Creative, diverse, sometimes incoherent
Perspective — Physics / Statistical Mechanics

Temperature here is exactly the Boltzmann distribution from statistical mechanics:

\[ p_i = \frac{e^{-E_i / kT}}{Z} \]

where logits are negative energies, and dividing by temperature flattens or sharpens the distribution. High temperature → all states equally likely. Low temperature → only the lowest energy (highest logit) state survives.

Top-k — Cutting Off the Tail

After temperature scaling, we keep only the top \(k\) logits and set the rest to \(-\infty\). This prevents the model from sampling very unlikely tokens that sneak through due to temperature.

Why both temperature AND top-k? Temperature makes the distribution flatter, which gives weird tokens a chance. Top-k removes the truly nonsensical ones. They work together: temperature for creativity, top-k for safety.

Concrete generation example
Prompt: "The cat sat"
Without temperature (T→0):  "The cat sat on the mat and looked around."
With temperature (T=0.8):   "The cat sat on the windowsill, watching the birds."
With high temp (T=1.5):     "The cat sat purple dreams through glassy time."

Low temp → safe, predictable continuations. High temp → creative but may be nonsensical. The "right" temperature depends on the use case.

"elephant" (token 3918) has logit -2.45 — very low. ──────────────────────────────────────────────────────── STEP 6: CROSS-ENTROPY LOSS ──────────────────────────────────────────────────────── For each of the 3 positions, we compute: Position 0: true token = 3797 ("cat") softmax prob of token 3797 in logits[0,0,:] = 0.52 loss[0] = -log(0.52) = 0.65 Position 1: true token = 6471 ("sat") softmax prob of token 6471 in logits[0,1,:] = 0.31 loss[1] = -log(0.31) = 1.17 Position 2: true token = 262 ("on") softmax prob of token 262 in logits[0,2,:] = 0.28 loss[2] = -log(0.28) = 1.27 total_loss = (0.65 + 1.17 + 1.27) / 3 = 1.03 ──────────────────────────────────────────────────────── END: loss = 1.03, ready for backward pass ────────────────────────────────────────────────────────
The key insight of the entire forward pass

Every single step is differentiable. The loss we computed (1.03) is a function of every parameter in the model — all 363 million of them. When we call loss.backward(), PyTorch computes the gradient of the loss with respect to every parameter using the chain rule. Then the optimizer adjusts each parameter to slightly reduce the loss next time.

Training is just repeating this forward pass + backward pass millions of times.

Perspective — Backpropagation Intuition

Think of backpropagation as the loss sending a message backward through every layer: "you (at this layer) contributed to the final error — here's how to adjust to reduce it."

The gradient at the logits is particularly clean: for the correct token, the gradient is \(p_{\text{correct}} - 1\) (negative → increase logit). For incorrect tokens, the gradient is \(p_{\text{wrong}}\) (positive → decrease logit). The model naturally learns to raise the correct answer's score and lower all others.

Part 6 — Key Takeaways

What is a logit?

A logit is a raw, unbounded score the model assigns to each vocabulary token. It's not a probability — it's an unnormalized vote. The model outputs 32,768 logits per position. Softmax converts them to probabilities.

How is loss computed?

For each position: softmax(logits) → probability of the correct token → \(-\log(p_{\text{true}})\). Average across all positions in the batch. At initialization: loss = \(\log(32768) = 10.40\). At convergence: ~3.0–3.5.

What is perplexity?

Perplexity = \(e^{\text{loss}}\). It answers: "How many tokens is the model effectively choosing among?" 30.6 means "as uncertain as choosing among ~31 equally-likely tokens." Much more interpretable than raw loss.

What does the forward pass look like end-to-end?

Input IDs → Embedding → 30× (RMSNorm → Attention → + → RMSNorm → MLP → +) → Final Norm → LM Head → Logits → (optionally) Cross-Entropy Loss. The shape stays \((B, T, 960)\) through the entire body, only changing at the embedding (token ID → vector) and LM head (vector → vocab scores).

Why does the model need all 30 layers?

Each layer refines the representation. Early layers learn syntax and local patterns. Middle layers learn semantics and coreference. Late layers learn higher-level reasoning. One layer can't do all of this at once — the depth lets the model build increasingly abstract representations.

Your next step to solidify understanding

Run the model on a single sequence. Print the shapes at every step. Print the logits for one position. Apply softmax manually and verify they sum to 1. Compute \(-\log(p_{\text{true}})\) manually and verify it matches the loss. There is no substitute for touching the actual tensors.
