FinalSLM · From Scratch

A simple but deep guide to building, training, and generating with a 363M-parameter decoder-only transformer

d_model = 960 Layers = 30 Heads = 15 Head dim = 64 Vocab = 32,768 Seq len = 2,048

The Big Picture

This model follows the Llama-style decoder-only architecture. That means it learns one job extremely well: given the tokens so far, predict the next token. Everything else in the system exists to make that job possible at scale: the tokenizer turns text into IDs, the data pipeline creates next-token examples, the transformer converts IDs into contextual vectors, the loss tells us how wrong the model was, and the optimizer nudges the weights so tomorrow's predictions are better than today's.

The easiest way to understand an SLM is not to memorize the code first. Follow the movement of information. Text becomes token IDs. Token IDs become vectors. Vectors move through 30 decoder layers. The final vector at each position becomes 32,768 scores, one score for every token the model could say next. Training repeatedly asks: did the correct next token receive a high score?

There are three fundamental dimensions that matter:

  • B — batch: how many independent sequences we process at once
  • T — time/position: where we are in the sequence (tokens in order)
  • d_model = 960 — the hidden / residual-stream width: what each token "knows"

Everything we do — attention, MLP, normalization — transforms the information inside each token's 960-dim vector while keeping the overall shape (B, T, 960) stable. The only two shape changes happen at the start (token IDs → vectors) and at the end (vectors → vocabulary scores).

⟹ The Central Insight

The residual stream (B, T, 960) is a highway that stays the same shape through all 30 layers. Each layer reads from it, transforms the information, and writes back. The shape never contracts or expands — only the content of the 960 features evolves.

⟹ What "from scratch" means in this repo

We are not writing CUDA kernels or reimplementing matrix multiplication. We are building the language model ourselves using PyTorch primitives: embedding lookup, linear layers, matrix multiply, softmax, cross-entropy, AdamW, and tensor reshaping. That is the right level for learning how an SLM works. The architecture, data packing, loss, training loop, saving, loading, and generation code are ours.

From-Scratch Build Map

If you wanted to build this SLM from a blank folder, this is the order. Each piece solves one concrete problem. Build them in this order because every later stage depends on the earlier one.

1. Tokenizer

Problem: neural nets cannot read raw strings.

Solution: train a 32,768-token byte-level BPE vocabulary in tokenizer/train_tokenizer.py.

Output: text like "The cat sat" becomes IDs like [464, 3797, 6471].

2. Data Stream

Problem: pretraining data is too large to load into memory.

Solution: stream FineWeb-edu, OpenWebMath, and StarCoderData in data/prepare_data.py.

Output: packed blocks of 2,049 tokens so input and labels can be shifted by one.

3. Model Config

Problem: architecture choices must be explicit.

Solution: define width, depth, heads, context length, and vocab size in configs/model_config.py.

Output: a 960-wide, 30-layer, 15-head decoder-only transformer.

4. Model Code

Problem: the model needs a forward pass.

Solution: implement RMSNorm, RoPE, attention, SwiGLU, decoder layers, tied LM head in model/slm_model.py.

Output: input_ids: (B,T) becomes logits: (B,T,32768).

5. Training Loop

Problem: random weights do not predict language.

Solution: run next-token cross-entropy, backprop, gradient accumulation, clipping, and AdamW in train/pretrain.py.

Output: weights slowly move from random guesses toward useful language predictions.

6. Evaluation

Problem: training loss alone can hide bugs.

Solution: compute validation loss, perplexity, and sample generations in eval/eval_loss.py.

Output: a basic answer to "is the model actually learning?"

⟹ Why these dimensions?

d_model=960 is divisible by 15 heads, giving head_dim=64. That 64-dim head size is a common transformer sweet spot: large enough for each head to learn a useful relation, small enough to keep attention efficient. intermediate_size=2560 is 8/3 × d_model, the common SwiGLU replacement for a standard 4x MLP. 30 layers makes the model deep enough to refine representations many times while staying around 363M parameters.

Layer-by-Layer Dimension Flow

Each section below follows the same pattern: intuition → what the dimensions mean → the math → the code → the output.

1 Token Embedding
(B, T) → (B, T, 960)

What happened? Each token ID (an integer 0–32767) looks up a row in a 960-column lookup table. This is the first time we go from "token labels" to "dense vectors the model can do math with."

⟹ What the dimensions mean
  • T — there are T positions in the sequence, each getting its own vector
  • 960 — each token is now represented by 960 numbers. At init these are random; during training they learn to encode meaning, syntax, and context.

Math: If \(E \in \mathbb{R}^{32768 \times 960}\) is the embedding matrix, then

\[ x_{t} = E[\text{token}_t] \qquad \text{for each position } t = 1, \dots, T \]

With our example: token[464] looks up row 464 of the embedding table. That row is a 960-dimensional vector. We do this for all 3 tokens. 

input_ids: (1, 3)
    ↓ embed(input_ids)
x: (1, 3, 960)   ← each of the 3 tokens now has a 960-dim vector
2 RMSNorm — Normalization
(B, T, 960) → (B, T, 960)

Same shape, but the 960 numbers per token are rescaled so their root-mean-square is ≈1. This prevents values from growing uncontrollably across 30 layers.

Why not LayerNorm? LayerNorm subtracts the mean first (centering). RMSNorm skips that step — it only scales. Empirically, the centering isn't needed for transformers, and RMSNorm is ~10–30% faster.

Math:

\[ \text{RMSNorm}(x) = \frac{x}{\sqrt{\frac{1}{d}\sum_{i=1}^{d} x_i^2 + \epsilon}} \cdot \gamma \]

Here \(\gamma\) is a learned vector of 960 values (initialized to all 1s). The model can learn to up-weight or down-weight specific features. 

# Step by step:
x.pow(2)              → (B, T, 960)    # square each element
.mean(dim=-1, keepdim=True) → (B, T, 1)  # average over 960 features
.rsqrt()              → (B, T, 1)    # 1 / sqrt(...)
x * rsqrt * gamma     → (B, T, 960)  # scale each token's vector
⟹ Why shape (B, T, 1) appears

When we compute the RMS, we average over the 960 features — one scale factor per token per position. The keepdim=True keeps the dimension so broadcasting works correctly when we multiply back with (B, T, 960).

3 RoPE — Rotary Position Embedding
inv_freq: (32,) → cos/sin tables: (2048, 32)

What problem does this solve? Attention is permutation-invariant — it doesn't know that "cat" comes after "The" but before "sat." We need to encode position.

The insight: Instead of adding a position vector (like absolute positional encoding), rotate the Query and Key vectors by an angle that depends on the position. The dot product \(Q_p \cdot K_q\) then naturally encodes the relative distance \((p - q)\).

How it works, step by step:

  1. Take the 64-dimensions in a head and pair them up: (0,1), (2,3), (4,5), …, (62,63) → 32 pairs
  2. For each pair \(i\), compute a base frequency: \( \mathrm{inv\_freq}[i] = 1 / \theta^{2i/64} \) where \(\theta = 10000\)
  3. For position \(p\), the rotation angle for pair \(i\) is: \(\theta_{p,i} = p \cdot \mathrm{inv\_freq}[i]\)
  4. Rotate each 2D pair by its angle: \(\begin{pmatrix} x'_{2i} \\ x'_{2i+1} \end{pmatrix} = \begin{pmatrix} \cos\theta & -\sin\theta \\ \sin\theta & \cos\theta \end{pmatrix} \begin{pmatrix} x_{2i} \\ x_{2i+1} \end{pmatrix}\)

What the precomputed tables look like:

# Cos table — first 8 positions, first 4 dimension-pairs:
pos 0: [1.000, 1.000, 1.000, 1.000, ...]   # at pos 0, rotation is 0°
pos 1: [0.999, 0.999, 0.999, 0.999, ...]   # at pos 1, tiny rotation
pos 2: [0.998, 0.997, 0.995, 0.988, ...]   # higher-frequency pairs rotate faster
pos 3: [0.995, 0.992, 0.983, 0.956, ...]
...
# Sin table — same structure, just sin instead of cos

Key point: The first pair (dim 0,1) rotates slowly — it encodes coarse position. Later pairs rotate faster — they encode fine-grained position. This is exactly analogous to how positional encoding works in the original Transformer (sine/cosine waves of different frequencies). 

# Before RoPE: Q and K are (B, 15, T, 64)
# RoPE splits into even/odd indices:
x_even = x[..., 0::2]    → (B, 15, T, 32)   # dims 0,2,4,...,62
x_odd  = x[..., 1::2]    → (B, 15, T, 32)   # dims 1,3,5,...,63

# Rotate each pair:
rot_even = x_even * cos - x_odd * sin   → (B, 15, T, 32)
rot_odd  = x_even * sin + x_odd * cos   → (B, 15, T, 32)

# Interleave back (stack → flatten):
stack → (B, 15, T, 32, 2) → flatten(-2) → (B, 15, T, 64)
⟹ Why only Q and K? Why not V?

The attention score is \(Q \cdot K^T\). This is where position matters — we need Q at position 1 (e.g., "The") to know its distance from K at position 3 ("sat"). The value V carries content that gets mixed regardless of position. Rotating V would change the content representation, which isn't what we want.

Visualizing the Rotation

Figure 2: RoPE rotates the Q (and K) vector by an angle proportional to position. Pairs of dimensions rotate as 2D vectors.
4 Multi-Head Causal Self-Attention
(B, T, 960) ⟶ (B, T, 960)

The purpose: Each token gathers information from all previous tokens (including itself). This is the mixing across positions step. The shape stays the same, but internally we split into 15 parallel "heads" that each learn different patterns.

Step-by-step dimension tour

Step 1 — Fused QKV projection:

x: (B, T, 960)
  ↓ @ W_qkv   (960 × 2880)
qkv: (B, T, 2880)         # fused: 960 for Q + 960 for K + 960 for V
  ↓ split into 3
q, k, v: each (B, T, 960)

Step 2 — Reshape into heads:

# Each of q, k, v goes through this:
q.view(B, T, 15, 64)   → (B, T, 15, 64)    # 15 groups of 64 dimensions
  .transpose(1, 2)      → (B, 15, T, 64)    # heads dimension moves to position 1

# Now q, k, v are each: (B, 15, T, 64)

Why 15 heads of 64? \(15 \times 64 = 960 = d_{\text{model}}\). Each head attends to the sequence independently — one head might track subject-verb agreement, another topic continuity, another punctuation patterns.

Note on the Transpose: Are you wondering why we `.transpose(1, 2)` the \(Q\) matrix here when the original mathematical formula \(\text{Attention}(Q, K, V)\) never transposes \(Q\)? See this detailed breakdown: The "Transpose of Q" in Multi-Head Attention.

Step 3 — Apply RoPE to Q and K (not V):

q ← apply_rope(q, cos, sin)   → (B, 15, T, 64)
k ← apply_rope(k, cos, sin)   → (B, 15, T, 64)
# v stays as (B, 15, T, 64) — no rotation

Step 4 — Attention scores:

scores = q @ k.transpose(-2, -1)   → (B, 15, T, T)
scores = scores / √64              # scale to keep softmax gradients healthy

Each of the 15 heads gets a T × T matrix where entry \((i, j)\) = "how much should token i attend to token j in this head?" The \(\sqrt{64}\) scaling prevents huge dot products (since 64-dim vectors have expected magnitude ~8).

Step 5 — Causal mask:

# Upper triangle (j > i) → -inf
# Token at position i can only attend to positions ≤ i
# This is what makes it "causal" or "auto-regressive"

scores[0, h, :, :]  # example for head h:
  [[2.1, -inf, -inf],     # token 0 attends to token 0 only
   [1.5, 0.8, -inf],      # token 1 attends to 0 and 1
   [0.3, 1.2, 0.6]]       # token 2 attends to 0, 1, and 2

Step 6 — Softmax + Weighted sum:

weights = softmax(scores, dim=-1)   → (B, 15, T, T)  # each row sums to 1
out = weights @ v                   → (B, 15, T, 64)

Each token's output is a weighted combination of the values of all tokens it can see. For token 2 ("sat"): 

out[2] = w[2,0]·v[0] + w[2,1]·v[1] + w[2,2]·v[2]
# where w[2,0] + w[2,1] + w[2,2] = 1
# e.g. [0.10, 0.67, 0.23] → "sat" looks mostly at "cat"

Step 7 — Concatenate heads + Output projection:

out: (B, 15, T, 64)
  ↓ .transpose(1,2)  → (B, T, 15, 64)
  ↓ .view(B, T, 960)   → (B, T, 960)    # all 15 heads back together
  ↓ @ W_o  (960 × 960)  → (B, T, 960)

The output projection mixes information across heads — the model can decide how to combine the 15 different perspectives into a single 960-dim representation.

⟹ The complete attention dimension flow

\((B, T, 960) \rightarrow\) QKV split \(\rightarrow 3\times(B, 15, T, 64) \rightarrow\) RoPE on Q,K \(\rightarrow\) scores \((B, 15, T, T) \rightarrow\) weights @ V \(\rightarrow (B, 15, T, 64) \rightarrow\) concat \(\rightarrow (B, T, 960)\)

The only point where T interacts with itself is the T×T attention matrix. Everything else is independent per position.

Visualizing Multi-Head Attention

Figure 3: Inside multi-head attention. QKV split → reshape into 15 heads → RoPE on Q,K → scores → causal mask → softmax → weighted sum → concat.
5 SwiGLU MLP
(B, T, 960) → (B, T, 960)

The purpose: Unlike attention (which mixes across positions), the MLP processes each position independently. It's the per-token "thinking" step — expanding to a larger intermediate space, applying a nonlinearity, and projecting back.

Why SwiGLU instead of a standard 2-layer MLP?

  • Standard MLP: \(y = W_2 \cdot \text{ReLU}(W_1 \cdot x)\) — 2 matrices
  • SwiGLU: \(y = W_{\text{down}}(\text{silu}(W_{\text{gate}} \cdot x) \odot W_{\text{up}} \cdot x)\) — 3 matrices

The gate path (\(\text{silu}(W_{\text{gate}} x)\)) learns to selectively activate or suppress features. The up path (\(W_{\text{up}} x\)) computes candidate features. The gate and up are multiplied element-wise — if the gate outputs near 0 for a feature, that feature is blocked. This gives the model finer control than a simple ReLU.

The math:

\[ \text{SwiGLU}(x) = W_{\text{down}} \big( \underbrace{\text{silu}(W_{\text{gate}} \cdot x)}_{\text{gate}} \odot \underbrace{W_{\text{up}} \cdot x}_{\text{candidate}} \big) \]

where \(\text{silu}(x) = x \cdot \sigma(x) = \frac{x}{1 + e^{-x}}\) (smooth, non-monotonic activation). 

x: (B, T, 960)

gate = silu(x @ W_gate)   → (B, T, 2560)   # gate path
up   = x @ W_up           → (B, T, 2560)   # up path
hidden = gate × up        → (B, T, 2560)   # element-wise gating
out  = hidden @ W_down    → (B, T, 960)    # project back down
⟹ Why intermediate_size = 2560?

A standard 2-matrix MLP with expansion factor 4 has intermediate size \(4 \times 960 = 3840\). With 3 matrices, to keep the same parameter count we use \(\frac{8}{3} \times d_{\text{model}} = 2560\). The parameter count: \(960 \times 2560 \times 3 \approx 7.37M\) compared to \(960 \times 3840 \times 2 \approx 7.37M\) — exactly the same.

6 The Decoder Layer — Putting It Together
(B, T, 960) → (B, T, 960)

One decoder layer = attention sublayer + MLP sublayer, each wrapped in pre-norm (normalize before the sublayer) and a residual connection (add the input back).

This is the most important architectural unit in the whole model. If you understand one decoder layer, the full 30-layer model is just this same operation repeated many times. The layer does not change the tensor shape. It changes the meaning stored inside the residual stream.

Figure 4: One decoder layer. Each sublayer proposes a correction; the residual path keeps the original stream available and keeps gradients stable.

The two equations that define a decoder layer:

\[ x' = x + \text{Attention}(\text{RMSNorm}(x)) \] \[ x'' = x' + \text{MLP}(\text{RMSNorm}(x')) \]

Why pre-norm? In the original "post-norm" Transformer, the normalization was applied after the sublayer. This caused gradients to vanish beyond ~12 layers. Pre-norm ensures the residual stream is always well-conditioned — gradients flow through the skip connection almost unchanged.

What question do these equations answer?

The question is: how can a deep model change information without destroying the useful signal already present? If a layer replaced x completely, every layer would risk wiping out what earlier layers learned. The residual form says: keep the old representation and only add a learned correction. That makes depth useful instead of fragile.

There are two kinds of correction. Attention says, "for this token, what should I borrow from earlier tokens?" That is why it builds the T × T matrix. MLP says, "now that this token has context, what internal features should I strengthen, suppress, or recombine?" That is why it expands 960 features to 2560, gates them, and compresses back to 960.

Parameter math inside one decoder layer

The layer shape stays \((B,T,960)\), but the learned matrices inside the layer are large. This is where most of the model's parameters live.

Component Weight shape Parameter count Why it exists
QKV projection 960 × 2880 2.76M Creates Q, K, and V in one fused matrix multiply.
Attention output projection 960 × 960 0.92M Mixes the 15 attention heads back into one residual update.
SwiGLU gate projection 960 × 2560 2.46M Decides which expanded features should pass through.
SwiGLU up projection 960 × 2560 2.46M Computes candidate features in the expanded space.
SwiGLU down projection 2560 × 960 2.46M Compresses the gated features back to the residual width.
Two RMSNorm scales 2 × 960 0.002M Learns per-feature scaling before attention and MLP.

One decoder layer is about 11.06M parameters. Thirty layers are about 331.8M parameters. The token embedding is another 32,768 × 960 = 31.5M. Because the LM head reuses that same embedding matrix, we avoid paying another 31.5M parameters at the output. This is how the full model lands around 363M parameters.

Same decoder layer from different perspectives

Perspective How to see the same layer
Architecture The layer has two jobs: attention mixes information across token positions; MLP mixes information across features inside each token.
Linear algebra Almost everything is matrix multiplication: project 960 features into Q/K/V, rotate Q/K pairs, multiply Q by K, then project back.
Optimization Residual connections make every layer learn a correction instead of a full replacement, which gives gradients a cleaner path backward.
Probability Attention weights are probabilities over visible tokens. Each row says how much this token should trust each previous token.
Information theory The layer moves useful context into each token's vector so the final distribution over next tokens has lower uncertainty.
Software / tensors The stable interface is the key engineering trick: every layer accepts (B,T,960) and returns (B,T,960).

Think of it this way: The residual stream is a "scratchpad" that starts as the token embedding. Each layer reads the current state, computes a correction (attention or MLP), and adds it back. After 30 corrections, the representation is rich enough to predict the next token.

7 Stacking 30 Layers

Each layer does the same thing: read from the residual stream, refine the representation, write back. But what they learn changes with depth:

  • Early layers (1–8): Learn syntax, word boundaries, basic grammar. Attention patterns are local (adjacent tokens).
  • Middle layers (9–20): Learn semantics, coreference, topic tracking. Attention spans grow longer.
  • Late layers (21–30): Learn higher-level reasoning, planning, and task-specific features. Attention becomes sparse and focused.

These boundaries are not hard rules. A trained transformer does not label one layer "grammar" and another layer "reasoning." This is the common pattern researchers observe: shallow layers tend to capture local and lexical features; middle layers combine context; later layers become more specialized for predicting the next token under the current task.

Figure 5: 30 decoder layers stacked in three groups. Each layer refines the (B, T, 960) representation. The shape never changes — only the content evolves.
Why repeat the same block 30 times?

One layer can do one round of communication and feature transformation. Thirty layers means thirty chances to revise the residual stream. A token can first detect local syntax, then connect to a subject, then resolve what phrase it belongs to, then prepare a next-token prediction. Depth gives the model an iterative computation, not just a bigger table.

From a math perspective, the stack is function composition: \(x_{1} = f_1(x_0)\), \(x_{2} = f_2(x_1)\), ..., \(x_{30} = f_{30}(x_{29})\). The key is that every \(f_i\) maps \((B,T,960)\) back to \((B,T,960)\). Because the interface is stable, we can stack many layers without changing the rest of the model.

8 Final Norm + LM Head — Making Predictions
(B, T, 960) → (B, T, 32768)

After 30 layers of refinement, we finally convert the model's understanding into predictions. This is where the rubber meets the road — we go from "what the model knows about each token" to "which token should come next." The feature dimension changes for the first time since the initial embedding.

Step 1: Final RMSNorm — One Last Stabilization

Before making predictions, we run RMSNorm one last time. This keeps the numbers well-behaved before the big projection. Same operation as before — just one final check. 

x: (B, T, 960)
  ↓ RMSNorm
x: (B, T, 960)   ← same shape, just stabilized values

Step 2: The LM Head — From "Meaning" to "Scores"

The LM Head (Language Model Head) is the final projection. It takes the 960-number "understanding" of each token and converts it into 32,768 raw scores — one for every token in the vocabulary.

This is where the architecture stops being "hidden representation" and becomes an actual language model. Before this point, every position has a 960-dimensional internal state. After the LM head, every position has a score for every possible next token. 

x: (B, T, 960)
  ↓ @ embed.weight.T  (matrix multiply: 960 × 32768)
logits: (B, T, 32768)   ← one raw score per vocabulary token
⟹ What the LM head actually does (very simply)

Think of the embedding matrix as a codebook with 32,768 entries. Each entry is a 960-number "meaning vector" for one token — like "the meaning of 'cat'" or "the meaning of 'sat'."

The LM head asks: "How similar is my current understanding (the 960-dim vector) to each of these 32,768 meaning vectors?"

Tokens whose meaning vector is similar get a high score. Tokens that are dissimilar get a low score (possibly negative).

Geometry View

At the last hidden layer, the model gives each position a vector \(h_t \in \mathbb{R}^{960}\). The tied LM head compares \(h_t\) with every token embedding \(E_i\). The logit for token \(i\) is roughly a dot product: \(z_i = h_t \cdot E_i\). If the angle and magnitude line up, the score is high. If they point in unrelated directions, the score is low.

What Are Logits? (The Most Important Concept)

Those 32,768 raw scores are called logits. Let's understand them from a few different angles:

Analogy — Voting

Imagine 960 experts voting on which token should come next. Each expert has their own opinion. The LM head counts all the votes. A logit of +5.1 means "lots of experts voted for this token." A logit of -3.2 means "most experts voted against this token."

Analogy — Thermometer

Think of logits as temperatures. A high positive logit = hot (the model feels strongly this token is right). A low negative logit = cold (the model feels this token is wrong). Zero = neutral.

Key Property — Logits Are Unbounded (Not Probabilities!)

Logits can be any number: -100, -5, 0, 3.7, 1000. There's no upper or lower limit. This is great for the model (it never has to hold back) but bad for human interpretation. We need softmax to convert them into probabilities.

Step 3: Softmax — Turning Raw Scores Into Probabilities

Softmax does two simple things:

  1. Exponentiate every logit: \(e^{\text{logit}}\). This makes all scores positive. A logit of 5.1 becomes ~164. A logit of -3.2 becomes ~0.04. Large logits stay large, negative logits become tiny.
  2. Divide by the total sum of all exponentiated logits. Now every number is between 0 and 1, and they all add up to 1.
\[ p_i = \frac{e^{z_i}}{\sum_{j=1}^{32768} e^{z_j}} \]

After softmax, we have a valid probability distribution over 32,768 tokens. High logit → high probability. Low logit → low probability. And \(\sum p_i = 1.0\).

Step 4: Cross-Entropy — How Training Judges the Prediction

Softmax gives a probability for every token. Training now asks one very simple question: how much probability did the model assign to the actual next token?

If the true next token is \(y\), the loss for this one position is:

\[ \text{loss} = -\log p_y \]

This equation exists because we need a penalty with the right behavior: if the model gives the correct token high probability, the loss is small; if it gives the correct token tiny probability, the loss becomes large. The logarithm matters because probability products over many tokens turn into sums, which makes training numerically manageable. 

# In model/slm_model.py:
loss = F.cross_entropy(
    logits.view(-1, logits.size(-1)),   # (B*T, vocab)
    labels.view(-1),                    # (B*T,)
)

# Meaning:
# every position in every sequence becomes one training example
Dry Run of the Loss

Suppose the correct next token after "sat" is "on". If the model gives \(p("on") = 0.28\), the loss is \(-\log(0.28) \approx 1.27\). If it gives \(p("on") = 0.001\), the loss is \(-\log(0.001) \approx 6.91\). The bad prediction receives a much stronger correction during backprop.

Concrete Example — "The cat sat"

At position 2 (predicting what comes after "sat"), the logits look like:

z["on"]  = 5.12   ← Very high! Model thinks "sat on" is likely
z["the"] = 3.78   ← Also high: "sat on the"
z["."]   = 2.01   ← Possible: end of sentence
z["sat"] = 0.84   ← Neutral: repeating the word?
z["cat"] = -1.15  ← Low: "cat" already appeared
z["elephant"] = -2.45 ← Very low: strange continuation

After softmax:
p["on"]  = 0.28   ← 28% chance (most likely)
p["the"] = 0.19   ← 19% chance
p["."]   = 0.06   ← 6% chance
p["sat"] = 0.04   ← 4% chance
p["cat"] = 0.004  ← 0.4% chance
p["elephant"] = 0.0003 ← 0.03% chance
(All 32768 probabilities sum to exactly 1.0)

The model thinks "on" is the best continuation (28% probability). If the actual next word in the data is "on", the model will be rewarded. If it's something else, the model will be penalized proportionally.

⟹ Tied Embeddings — Saving 31.5 Million Parameters

The same matrix \(E\) (32,768 × 960) does double duty:

  • At the start (Embedding): Token ID → look up row in \(E\) → get the 960-number meaning vector
  • At the end (LM Head): Meaning vector → multiply by \(E^T\) → get 32,768 vocabulary scores

Why does this work? The embedding converts token → meaning. The LM head converts meaning → token. They're approximate inverses of each other, so sharing the matrix makes sense. This saves \(32768 \times 960 = 31.5\) million parameters that would otherwise be wasted on a separate LM head.

9 Generation — From Logits to Tokens

During training, we use all T positions (each one predicts the next token). But during generation, we only need one thing: what token comes after everything we've seen so far?

logits[:, -1, :] → (B, 32768)

Why Only the Last Position?

Think about reading a sentence: "The cat sat".

  • Position 0 ("The") predicts token 1 ("cat"). But we already know the next word is "cat".
  • Position 1 ("cat") predicts token 2 ("sat"). We already know the next word is "sat".
  • Position 2 ("sat") predicts the word that comes after "sat". This is the only one we don't know yet.

So we take logits[:, -1, :] — the logits at the last position — and use those to predict what comes next.

The Generation Loop — Predict, Append, Repeat

Generation is a loop. At every step:

  1. Feed the entire sequence (original prompt + any new tokens) into the model
  2. Look at the last position's logits
  3. Apply temperature, top-k, softmax, and sample one token
  4. Append the new token to the sequence
  5. Repeat until we hit the desired length or the model says "stop"
# Step-by-step in code:
for _ in range(max_new_tokens):

    # 1. Run the model on everything we have so far
    logits = model(input_ids).logits              # (B, T, 32768)

    # 2. Take ONLY the last position's logits
    next_logits = logits[:, -1, :]               # (B, 32768)

    # 3. Temperature scaling (creativity dial)
    next_logits = next_logits / temperature        # e.g. T=0.8

    # 4. Top-k filtering (cut off bad options)
    top_k = 50
    values, _ = torch.topk(next_logits, top_k)
    next_logits[next_logits < values[:, -1:]] = float('-inf')

    # 5. Softmax → probabilities, then sample one token
    probs = F.softmax(next_logits, dim=-1)       # (B, 32768)
    next_id = torch.multinomial(probs, 1)          # (B, 1)

    # 6. Append the new token, go back to step 1
    input_ids = torch.cat([input_ids, next_id], dim=1)
Figure 6: The generation loop — take last logits, apply temperature + top-k, softmax, sample, append, repeat.
Training vs Generation

During training, the model predicts every position in parallel because the whole sequence is already known. During generation, only the next token is unknown, so we use only logits[:, -1, :]. After sampling one token, that token becomes part of the input, so the sequence length grows from \(T\) to \(T+1\).

The generation math is the same LM-head math as training, but used one step at a time. For the current final position, the model computes logits \(z \in \mathbb{R}^{32768}\). Temperature changes those logits to \(z / \tau\). Top-k sets all but the largest \(k\) logits to \(-\infty\). Softmax turns the remaining scores into probabilities. Sampling picks one token ID from that probability distribution. 

# One generation step as math-shaped code:
z = logits[:, -1, :]          # (B, vocab)
z = z / temperature           # sharpen or flatten the distribution
z[not_in_top_k] = -inf        # remove low-quality candidates
p = softmax(z, dim=-1)        # probabilities over vocabulary
next_id = sample(p)           # (B, 1)
input_ids = concat(input_ids, next_id)

Understanding Temperature — The Creativity Dial

Temperature controls how much randomness the model uses when choosing the next token. Here's what it does:

TemperatureWhat it does to the scoresHow the model behaves
Low (0.1 — 0.5) Dividing by a small number makes the big logits HUGE and the small logits TINY. The gaps get wider. Safe, repetitive, predictable. Model always picks the most obvious token. Like giving the textbook answer.
Medium (0.7 — 1.0) Normal. Whatever the model learned during training. Natural and balanced. Model sounds like a normal person. Default setting for most uses.
High (1.2 — 2.0) Dividing by a bigger number makes all logits closer together. Low-probability tokens get a real chance. Creative, surprising, sometimes nonsense. Model takes risks. Good for brainstorming. May produce garbage.

▸ Simple Analogy — The Creativity Dial

Low temp (0.3): The model plays it safe. It says the most predictable thing every time. Like answering an exam — you give the standard correct answer.

High temp (1.5): The model gets creative. It might say something surprising. Like brainstorming ideas in a group — some are brilliant, some are weird.

You choose the temperature based on what you need: facts and accuracy → low temp. Creativity and variety → high temp.

▸ How Temperature Changes the Math

Softmax with temperature: \(p_i = \dfrac{e^{z_i / T}}{\sum_j e^{z_j / T}}\). Dividing by \(T\) before exponentiating. When \(T\) is small (e.g. 0.3), \(z_i / T\) becomes a bigger number → the exponent widens the gap between high and low tokens → the top token gets almost all the probability mass. When \(T\) is large (e.g. 1.5), \(z_i / T\) shrinks → all probabilities become more equal → more randomness in choosing.

Understanding Top-K — Cutting Off Bad Options

After temperature scaling, we can do top-k filtering: we look at all 32,768 tokens, keep only the top \(k\) with the highest logits, and set the rest to \(-\infty\) (they become impossible to pick).

Why? Even with temperature, some truly weird tokens might sneak through. Top-k chops them off entirely. With top-k=50, the model only considers the 50 most likely tokens and ignores the other 32,718. This prevents the model from randomly picking a byte value or a rare token that doesn't make sense.

Common values: k=40 to k=100. Lower = safer, higher = more variety.

Understanding Sampling — Why We Don't Always Pick #1

After softmax, we have a probability distribution over 32,768 tokens. We could just pick the token with the highest probability (called greedy decoding or argmax).

But greedy decoding makes repetitive, boring text. The model always picks the single most obvious word, which leads to loops like "I love love love love love..."

Instead, we sample — we roll a 32,768-sided weighted die. The die is weighted by the probabilities:

  • The most likely token has the best chance of being picked (but isn't guaranteed)
  • The second most likely has a decent chance
  • The 50th most likely has a small chance
  • The 32,768th most likely has virtually no chance (especially with top-k)

This makes the generated text natural and varied, just like how humans don't always say the single most predictable word. Sometimes we pick the second-best word for variety.

⟹ The Big Pattern

Steps that change shape: 1→2 (embedding), 4 (QKV expansion), 6 (reshape heads), 8 (T×T scores), 11 (concat heads), 18 (LM head projection). Everything else is a same-shape transformation that refines the content without changing the structure.

Key Takeaways

  1. The residual stream is invariant. Its shape \((B, T, 960)\) never changes from embedding to final norm. All 30 layers read from and write to it without altering its dimensions.
  2. Attention = mixing across positions. The only place where the T dimension interacts with itself is the \(T \times T\) attention matrix. Each head computes its own attention pattern.
  3. MLP = mixing across features. The MLP operates on each position independently. It expands to a wider space (2560), applies a gated nonlinearity, and projects back to 960. No cross-position communication.
  4. Two shape changes: At the start (embedding: IDs → vectors) and at the end (LM head: vectors → vocab scores). Everything in between is \((B, T, 960) \rightarrow (B, T, 960)\).
  5. Heads = parallel attention views. \(d_{\text{model}}\) is split into 15 heads of 64 dimensions. Each head runs independent attention. The output projection lets the model recombine them.
  6. RoPE encodes position without changing shape. It rotates Q and K vectors so the dot product encodes relative distance. Applied only to Q and K, not V.
  7. Pre-norm + residual = trainable depth. Normalizing before each sublayer and adding skip connections lets gradients flow through 30 layers without vanishing.
  8. Generation uses only the last position. During inference, we take logits[:, -1, :], apply temperature/top-k, softmax, and sample one token. That token is appended and the process repeats.
